# 微信小程序迁移

## Problem Statement

Fishing Buddy（渔游记）当前使用 Flutter 跨平台框架，目标用户为中国国内钓鱼爱好者。核心矛盾：Flutter App 需要用户下载安装，传播路径长，获客成本高。微信小程序无需安装，可直接转发给好友/群聊，天然适合社交传播场景。用户希望通过微信生态获取新用户，而非应用商店。

## Solution

将技术栈从 Flutter 迁移到原生微信小程序（WXML/WXSS/JS），使用微信云开发（CloudBase）作为后端。保留核心钓鱼指数算法，渐进式实现功能：MVP（指数 + 记录 + 历史）→ B 阶段（图表 + 分享 + 设置）。

## User Stories

1. 作为钓鱼爱好者，我想打开微信小程序就能看到今天适合不适合钓鱼（钓鱼指数），这样我不用出门就知道今天值不值得去
2. 作为钓鱼爱好者，我想看到未来 7 天每天的钓鱼指数评分，这样我可以提前规划周末的钓鱼行程
3. 作为钓鱼爱好者，我想看到钓鱼指数的 8 个因素各自得分（气压、风力、温度、天气、月相、节气、昼夜温差、溶氧量），这样我能理解为什么今天适合或不适合钓鱼
4. 作为钓鱼爱好者，我想看到每天的具体天气数据（气温范围、风速、天气状况、气压），这样我能判断是否需要带雨具
5. 作为钓鱼爱好者，我想在小程序里记录一次钓鱼行程（出发时间、钓点、鱼种、数量），这样我不用纸笔记
6. 作为钓鱼爱好者，我想记录行程时自动获取我的 GPS 定位并自动填入钓点名（通过反向地理编码），这样我不用手动输入经纬度
7. 作为钓鱼爱好者，我想记录行程时选择鱼种（从 80-100 种常见鱼中选），这样我不用每次手打鱼名
8. 作为钓鱼爱好者，我想记录行程时填写饵料类型、装备类型、最大鱼长度、总重量、备注，这样我的记录更完整
9. 作为钓鱼爱好者，我想记录没有钓到鱼的行程（空军），这样我的统计数据准确反映真实情况
10. 作为钓鱼爱好者，我想查看所有历史行程列表，按时间倒序排列，这样我能快速找到某次钓鱼记录
11. 作为钓鱼爱好者，我想在历史列表中看到每次行程的摘要（钓点、时间、鱼种、数量、是否空军），这样不用点进去就能了解概况
12. 作为钓鱼爱好者，我想点击历史行程查看详情（完整信息：时间、钓点、装备、鱼种列表、天气、备注），这样我能回顾那次钓鱼的全部信息
13. 作为钓鱼爱好者，我想在记录行程时，如果 GPS 定位在 100m 内有历史钓点，自动匹配并使用已有钓点名，这样我的钓点数据不会因为手填而产生重复
14. 作为钓鱼爱好者，我想把小程序页面转发给微信好友，这样他们也能看到今天的钓鱼指数
15. 作为钓鱼爱好者，我想转发时自定义分享卡片的标题和缩略图（比如"今天钓鱼指数 85，强烈建议出行"），这样分享内容更有吸引力
16. 作为钓鱼爱好者，我想在 B 阶段看到出钓频率柱状图（按月），这样我能了解自己钓鱼的频率趋势
17. 作为钓鱼爱好者，我想在 B 阶段看到鱼种分布饼图，这样我能知道自己最常钓到什么鱼
18. 作为钓鱼爱好者，我想在 B 阶段看到渔获趋势折线图，这样我能了解自己的钓鱼水平是否在提升
19. 作为钓鱼爱好者，我想在 B 阶段看到中鱼率百分比，这样我能知道自己的成功率
20. 作为钓鱼爱好者，我想在 B 阶段切换重量单位（斤/kg），这样我习惯用哪个就用哪个
21. 作为钓鱼爱好者，我想在小程序里完全不需要登录或注册，打开就能用，这样没有使用门槛

## Implementation Decisions

### Architecture

- **微信小程序原生开发**: WXML + WXSS + JS，不使用 Taro/uni-app 等跨端框架
- **微信云开发（CloudBase）**: 免运维后端，自带用户鉴权（openid），云数据库 + 云函数 + 云存储
- **嵌套文档结构**: Trip 文档内嵌 species 数组和 weather 对象，Spot 信息冗余嵌入 Trip，避免二次查询

### 目录结构

```
fishing-buddy-miniprogram/
├── miniprogram/
│   ├── app.js / app.json / app.wxss
│   ├── pages/ (home, record, history, detail)
│   ├── components/ (fishing-index, forecast-card, trip-card, species-picker, bait-picker)
│   ├── utils/ (fishing-index.js, haversine.js, species-data.js, bait-data.js, date-utils.js)
│   └── images/
├── cloudfunctions/
│   ├── weather-proxy/ (OpenWeatherMap 代理)
│   └── reverse-geocode/ (腾讯地图反向编码)
└── project.config.json
```

### Navigation

- 3 Tab 结构: 首页（合并钓鱼指数 + 7天预报 + 最近行程）、记录、历史
- 行程详情页为子页面跳转（非 Tab）
- 首页是核心价值入口，打开即见钓鱼指数

### Data Storage

- **云数据库集合 `trips`**: 嵌套文档，每条 Trip 包含嵌套的 species 数组和 weather 对象，Spot 信息（name、lat、lon）冗余嵌入
- **索引**: `_openid + startTime`（用户历史列表查询）、`spot.latitude + spot.longitude`（钓点去重查询）
- **用户隔离**: 所有查询通过 `_openid` 自动过滤，用户只能看到自己的数据

### Weather API

- **OpenWeatherMap** `/data/2.5/forecast`（免费 1000 次/天，提供气压 pressure_hpa 数据）
- **云函数代理**: API Key 存云函数环境变量，客户端不暴露密钥
- **返回结构**: 映射为 `DayWeatherForecast` 模型（date、airTemp、minTemp、maxTemp、pressureHpa、windSpeedMps、weatherCondition、hasRainForecast）

### Fishing Index Algorithm

- 8 因子评分逻辑从 Flutter 版本 1:1 移植为 JavaScript
- 每因子 0-20 分，总分 0-100
- 推荐等级: >=80 强烈建议、>=60 建议、>=40 可以尝试、<40 不建议
- 纯计算逻辑，无外部依赖

### GPS / Location

- `wx.getLocation` 获取经纬度（需用户授权）
- 腾讯地图 WebService API 反向地理编码（免费 5000 次/天）
- Haversine 公式 100m 阈值钓点去重
- MVP 阶段钓点名手填，GPS 仅获取经纬度；完整版加反向编码自动命名

### Fish Species / Bait Database

- MVP 阶段前端硬编码 80-100 种常见淡水+海水鱼列表，零网络开销
- 饵料列表同理硬编码
- 后续版本迁移至云数据库支持用户自定义增删

### Charts

- wx-charts（纯 Canvas，~50KB），B 阶段用于出钓频率柱状图、鱼种分布饼图、渔获趋势折线图
- 不使用 ECharts（~300KB+，包体积过大）

### Social Sharing

- `onShareAppMessage` 基础转发能力
- 自定义分享卡片标题和缩略图
- MVP 阶段实现基础转发，B 阶段加自定义卡片

### Unit System

- 默认重量单位: 斤（1 斤 = 500g）
- B 阶段支持切换至 kg
- 存储内部统一用 kg，展示层转换

## Testing Decisions

### What makes a good test

- 测试外部行为，不测试实现细节。例如测试"气压 1010 hPa 时钓鱼指数为 X"而非测试内部函数调用
- 云函数测试用 mock HTTP 请求，不真实调用第三方 API
- 页面逻辑测试验证数据流和用户交互结果

### Test layers

1. **Fishing index algorithm** (单元测试) — 纯 JS 计算逻辑，100% 覆盖率，包括所有 8 因子的边界条件（气压 1005/1020、风速 1/5、温度 15/25 等）
2. **Cloud functions** (单元测试) — Mock HTTP 请求测试 weather-proxy 和 reverse-geocode，验证成功/失败/超时场景
3. **Page logic** (集成测试) — 验证数据流：获取天气 → 计算指数 → 渲染首页、提交表单 → 写入数据库 → 列表展示
4. **End-to-end** (端到端测试) — 完整流程：记录行程 → 查看历史 → 查看详情

### Prior art

- Flutter 版本 PRD 规划了 4 层测试但未实现，小程序版本从零开始建立测试体系
- 钓鱼指数算法是核心价值，必须有完整测试覆盖

## Out of Scope

- 跨端框架（Taro、uni-app）— 使用原生微信小程序
- 离线能力 — 小程序天然依赖网络
- 持续后台 GPS 定位 — 仅前台单次获取
- 自定义鱼种/饵料（MVP 阶段）— 硬编码列表
- 数据导出（CSV/JSON）— 后续版本
- 分享海报图生成（C 阶段）— B 阶段只做转发卡片
- 多语言支持 — 仅中文
- 用户账号系统 — 使用微信 openid 自动绑定
- 关系型数据库 — 使用云开发 MongoDB 风格文档数据库
- Flutter 版本的保留或维护 — 全新重写

## Further Notes

### 包体积约束

微信小程序主包限制 2MB，分包单个 2MB，总包 20MB。wx-charts ~50KB 可接受，ECharts ~300KB+ 不可接受。

### 天气 API 数据源变更

原 Flutter 版本使用和风天气（`and_feng_weather_client.dart`），迁移到 OpenWeatherMap。CONTEXT.md 中 Trip Attributes 的天气来源标记需同步更新。

### 云开发免费额度

微信云开发免费额度：数据库存储 2GB、云函数调用 20万次/月、云存储 5GB。个人项目足够，无需付费。

### GPS 授权体验

wx.getLocation 需要用户授权，首次使用弹授权弹窗。建议在用户点击"获取定位"按钮时触发，而非页面加载时自动触发，避免转化率损失。
